 For closure, you already understand the main idea: an inner function can keep access to variables from an outer function
  even after the outer function has finished.

  What you still lack:

  1. Difference between returning a value and returning access to a variable

  In closure_1.js, you returned count directly:

  return { increment, count };

  That gives the outside code the value of count at that moment, not a live private variable.

  You meant:

  return { increment, getCount };

  This is important: closure works when a function keeps access to the variable, not when you copy the variable’s value out.

  2. Closure with var vs let in loops

  Your closure loop exercise uses let, so it already works:

  for (let i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 100);
  }

  To understand the classic closure problem, you need to study this version:

  for (var i = 0; i < 3; i++) {
    setTimeout(() => console.log(i), 100);
  }

  This prints:

  3
  3
  3

  You should understand why var shares one function-scoped variable, while let creates a new block-scoped binding for each
  loop iteration.

  3. Closure is about variables, not just nested functions

  A nested function is not the full concept. The key idea is that the inner function remembers the outer variable.

  This is closure:

  function makeAdder(x) {
    return function (y) {
      return x + y;
    };
  }

  x stays alive because the returned function still needs it.

  4. Multiple closures can share the same private state

  You did this with account/game examples, but you should name the idea clearly:

  return { deposit, withdraw, getBalance };

  All three methods share the same closed-over _balance.

  5. Each function call creates a new closure

  You should practice this more:

  const counter1 = createCounter();
  const counter2 = createCounter();

  counter1(); // 1
  counter1(); // 2
  counter2(); // 1

  Each call to createCounter() creates a separate count.

  6. Closures can cause memory retention

  If a returned function keeps access to a large object, that object cannot be garbage-collected yet. This is more advanced,
  but important later.

  ———

  For event loop, you already understand that synchronous code runs first, promises run before timers, and async callbacks
  run later.

  What you still lack:

  1. Precise microtask vs macrotask ordering

  You know promises run before setTimeout, but you need a sharper model:

  1. Run all synchronous code
  2. Run microtasks
  3. Run one macrotask/task
  4. Run microtasks again
  5. Repeat

  Examples of microtasks:

  Promise.then()
  queueMicrotask()
  await continuation

  Examples of macrotasks/tasks:

  setTimeout()
  setInterval()
  setImmediate() // Node-specific
  I/O callbacks

  2. Node has process.nextTick()

  In Node.js, process.nextTick() runs before Promise microtasks.

  So this:

  console.log("start");

  process.nextTick(() => console.log("nextTick"));
  Promise.resolve().then(() => console.log("promise"));

  console.log("end");

  Outputs:

  start
  end
  nextTick
  promise

  Your comment in event_loop/exercise_5.js had the order wrong.

  3. setTimeout(..., 0) does not mean immediate

  It means “run after the current call stack is empty and after microtasks finish.”

  So:

  setTimeout(() => console.log("timeout"), 0);
  Promise.resolve().then(() => console.log("promise"));
  console.log("sync");

  Outputs:

  sync
  promise
  timeout

  4. setTimeout vs setImmediate is Node-specific and context-dependent

  At top level, order can vary:

  setTimeout(() => console.log("timeout"), 0);
  setImmediate(() => console.log("immediate"));

  But inside I/O, setImmediate usually runs first:

  fs.readFile(__filename, () => {
    setTimeout(() => console.log("timeout"), 0);
    setImmediate(() => console.log("immediate"));
  });

  Usually outputs:

  immediate
  timeout

  5. await continues in a microtask

  This:

  async function foo() {
    console.log("1");
    await null;
    console.log("2");
  }

  console.log("3");
  foo();
  console.log("4");

  Outputs:

  3
  1
  4
  2

  You noted that await pauses, but you should also connect it to the microtask queue.

  6. Long synchronous code blocks the event loop

  Your event_loop_1.js touches this, but the idea should be clearer:

  setTimeout(() => console.log("timer"), 0);

  while (Date.now() < start + 3000) {}

  console.log("done");

  The timer cannot run until the blocking loop finishes.

  Event loop:
